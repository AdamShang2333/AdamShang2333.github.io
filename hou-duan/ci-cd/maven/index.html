#Maven

自动化的构建工具

##1. Maven 的作用
1 管理 Jar 包

a) 增加第三方 Jar 包
　　 通常，先在本地仓库进行下载，如果没有，先去私服仓库 (局域的仓库); 再联网从中央仓库 (网络仓库) 下载
b) Jar 包之间的依赖关系 (自动把 A jar 包需要的 B C 包一起关联下载)

2 将项目拆分成若干模块
###maven 文件目录结构
####src

放源代码

####target

放编译后的文件

####pom.xml

配置文件

### 功能模块:
clean

清理：删除编译的结果 (删除 target 目录), 为重新编译做准备

validate

compile

编译 : Java --> class (只编译 main 目录中的 java 文件，编译后的文件会放在 target 文件夹中)

test

测试:

package

打包：将文件自动压缩成一个 war 文件，用于安装或部署

verify

install

安装：将打包的文件，放在本地仓库 (存放 Jar 包等资源的地方), 供其他项目使用

site

deploy

部署：将打包的文件，放在服务器上准备运行

##3. 依赖

管理 Jar 的时候，通过依赖来明确 jar 包的耦合管理

<dependencies>
	<dependency>
      <groupId>commons-dbutils</groupId>
      <artifactId>commons-dbutils</artifactId>
      <version>1.6</version>
    </dependency>
    
    <dependency>
      <groupId>junit</groupId>
      <artifactId>junit</artifactId>
      <version>4.11</version>
      // 标识依赖的范围和有效性
      <scope>compile</scope>
    </dependency>
    ...
    ...
<dependencies>

###3.1 依赖的范围，有效性

Maven 在编译 测试 部署 (运行) 时，各自使用一套 classPath

compile (编译) ---- 默认值

test (测试)

provided (准备)
---->Servlet-api.jar

#### 依赖的有效性

通俗的理解，就是在不同的阶段，能不能拿到 jar 包

| 运行阶段 | compile|test|provided|
|:--😐:--😐:--😐:--😐:
| 编译 |✔️|❌|✔️|
| 测试 |✔️|✔️|✔️|
| 部署 (运行)|✔️|❌|❌ |

###3.2 依赖的排除

如果自动导入的 jar 包，有关联的依赖 jar 包，maven 会自动下载
但是使用的时候不需要自动下载的关联 jar 包的时候，
可以使用依赖排除，来手动取消关联 jar 包的下载

IDEA排除依赖, 在关系视图中选择 "exclude" 选项后, 刷新即可;会自动生成排除的代码

  <dependency>
       <groupId>junit</groupId>
       <artifactId>junit</artifactId>
   	   <version>4.11</version>
   	   <scope>compile</scope>
   	 // 排除依赖
   	 <exclusions>
     	<exclusion>
       	   <artifactId>hamcrest-core</artifactId>
       	   <groupId>org.hamcrest</groupId>
        </exclusion>
      </exclusions>
  </dependency>

###3.3 依赖的传递性

A.jar --> B.jar --> C.jar
要使: A.jar --> C.jar
则需要：当且仅当 B.jar 依赖于 C.jar 的范围是 compile

###3.4 多个 maven 项目之间如何依赖
P 项目 --> Q 项目
步骤

  1. P 项目 install 到本地仓库
  2. Q 项目依赖 P 项目:
<dependency>

  <groupId>P项目文件名(com.lanou.X X X)</groupId>
  <artifactId>P项目的项目名(Y Y Y)</artifactId>
  <version>1.6</version>

</dependency>

###3.5 依赖原则
目的: 防止依赖冲突

####1 路径最短优先原则

自身 POM.XMl 有一个依赖的 jar 包；
也可以通过依赖传递获取 (A-->C), 选择自身 pom.xml 中的

####2 路径长度相同

i . 在同一个 Pom 文件中，有 2 个相同的依赖，后声明的依赖会覆盖前面声明的依赖
一般来说，禁止在同 1 个 Pom 中依赖两个同名的
ii . 如果是不同的 Pom 中，有 2 个相同的依赖，则先声明的依赖会覆盖后声明的依赖
　即: C 先依赖于 A 后依赖于 B , 二者都有 X.jar ; 那么 C 依赖 A 中的 X.jar
　　---> 取决于 C 依赖于 A,B 2 个 Pom 的代码顺序

##4. 继承

A --> B, 则 A 可以使用 B 的所有依赖

解决了依赖传递时要求的依赖范围必须是compile的约束

继承实现步骤

##maven 生命周期
### 生命周期和构建的关系

生命周期中的顺序: a , b , c , d , e ... 命令
当我们执行 c 命令的时候，之前的 a , b 命令都会先执行一遍
即：实际执行了 a , b c 三个命令

生命周期包含三个阶段:
	1.  clean lifecycle     -- 清理
			pre-clean  clean  post-clean
	2.  default lifecycle   -- 默认(常用)
	3.  site lifecycle		-- 站点
			pre-site   site   post-site  site-deploy
