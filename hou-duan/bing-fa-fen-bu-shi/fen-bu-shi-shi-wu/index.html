# 分布式事务

# 分布式事务的产生

传统的单体架构下，数据库并不会产生分离。对于事务的概念和控制，都是针对数据在同一个数据库下。
而在微服务的架构下，数据库为了支持高并发，往往都会进行数据的分库分表或者读写分离的设计，这时对于事务的控制，需要以跨数据库为前提进行实现。由此就产生了分布式的事务问题。

产生分布式事务的本质原因，就是数据库的操作是跨库

# 分布式事务的解决方案

# 两阶段提交（TCC）

两阶段提交，需要一个事务管理器（中间协调者）：

  1. 若干数据库的 sql 执行结果都提交到事务管理器上（第一阶段 预提交）。在第一阶段，会预先锁定一些数据库的资源，预先校验数据库，缓存等服务是否正常。
  2. 各个数据库执行结果均正常，事务管理器通知数据库提交 sql 操作；
    若有任一数据库操作失败，所有数据库的操作回滚（第二阶段 真正提交 / 回滚）

实际上，分布式事务时不可能做到 100% 的解决，只能尽量提高可靠性

到达事务管理器之前的数据库宕机之类，对整个事务不会产生问题：直接按照失败 rollback;
问题主要是在数据库预提交都成功后

提高分布式事务可靠性
1. 事务管理器通知数据库提交时，可能产生通知失败（消息丢失或数据库宕机）。这种情况下，可以对事务管理器添加重试机制
2. 事务管理器在接收和发送事务提交通知时宕机，可以对事务管理器添加 log 日志 在宕机重启后人工介入重新提交

# 微服务架构下的分布式事务

# TCC（try-confirm-cancle） 两阶段补偿型方案

在微服务的背景下，单体应用一般都会拆分成各个服务模块。此时分布式事务不仅跨库，也跨服务。无法使用普通的 2pc 在一个后端服务中， 对多个数据库进行预提交到事务管理器并处理提交的逻辑。

解决方案：
与 2pc 的预提交阶段思想一致，为了提高可靠性，尽量的在真正提交之前预先验证和锁定资源会很大程度上提高事务的可靠性。
所以在微服务架构下，可以将调用的其他服务添加一个 "预提交" 状态的 API。
例：支付系统的订单服务，调用库存 积分 仓储等其他服务模块

阶段 1: try （预留资源，中间状态）
订单服务修改状态为更新中，而不是下单成功或失败；
API 调用其他服务，分别置为 "冻结库存"，"预加积分"，"创建出库单草稿" 等 预留资源状态，以表示尝试进行分布式事务 但执行结果不确定的状态
阶段 2 ： 确认资源（confirm/cancel）
在第二阶段，订单服务根据其他服务调用的执行反馈，确定是整个事务成功提交，还是对事务进行回滚。因为现在使用微服务架构，所有的操作都需要 RPC 调用完成，这里需要根据 conform 和 cancel 编写两套 API 到各个子服务，以支持 TCC 的分布式事务实现。

数据库表结构上，需要在原有的库存字段等基础上，添加一列 "冻结" 字段，来表示第一阶段预留资源，分别在成功 / 回滚的逻辑下，结合现有库存字段来完成订单支付成功 / 失败的一系列库存 积分等数据更改。

在并发访问量不是很高的情况下 2pc 和 TCC 的性能差异不会很大，甚至 2pc 的速度还可能更快一点（不需要 RPC 调用若干服务）。
但是访问量比较高的情况下 ，TCC 的性能是要远远优于 2PC 的。因为在高并发情况下，2PC 的方案会对整个数据库 "加锁" 一直占用资源直到事务管理器通知提交或回滚；而 TCC 的方案 在调用其他服务时，实际上在各自的数据库中直接更改并提交了数据。不会在全局一直占用资源
