# ZooKeeper 入门

# 1 概述

是一个开源的分布式的，为分布式框架提供协调服务的 Apache 项目

# 工作机制

基于观察者模式设计的分布式服务管理框架，负责存储和管理数据， 然后接受观察者的注册，一旦数据发生变化，ZooKeeper 就会通知已经在 ZooKeeper 上注册的那些观察者做出响应

# 特点

  1. 是由 1 个领导者 (Leader) , 多个跟随者 (Follower) 组成的集群
  2. 集群中只要有半数以上节点存活，集群就能正常服务 ==> ZooKeeper 适合安装奇数台服务器
  3. 全局数据一致：每个 server 保存的数据副本都相同
  4. 更新请求顺序执行：来自同一个 Client 的更新请求按其发送顺序一次执行
  5. 数据更新原子性

# 数据结构

数据模型的结构与 Unix 文件系统很类似，整体上可以看做是一颗树，每个节点称作一个 ZNode . 每一个 ZNode 默认能够存储 1MB 数据， 每个 ZNode 都可以通过其路径唯一标识

# 应用场景

  • 统一命名服务
    • 对服务进行统一命名，便于识别，交给 ZooKeeper 维护服务对应的多台服务器的 ip (Nginx 亦可实现)
  • 统一配置管理
    • 注册到 ZooKeeper 中的集群，节点配置信息都是一致的
    • 配置文件修改后，能够快速同步到各个节点
      • 配置信息写入到 ZNode 上，由 Zookeeper 监听，一旦数据被修改，ZooKeeper 将通知各个客户端服务器
  • 统一集群管理
    • 将集群节点的信息写入 ZNode, 监听 ZNode 即可获取它的实时状态变化

  • # 服务器动态上下线

  • 软负载均衡
    • 根据节点的服务器压力，动态选择合适的服务器 (Nginx 亦可)

# 服务端基础配置参数

Zookeeper 的配置文件在 zoo.cgf 中

常用的几个配置参数如下:

  • tickTime = 2000
    • 通信心跳时间，Zookeeper 服务器与客户端心跳时间 (周期性的发送一些信息), 单位毫秒.
  • initLimit = 10 (10 代表 10 个 tickTime)
    • LF 初始化通信时限 (Leader 和 Followe 初始连接时能容忍的最多心跳数)
  • syncLimit = 5 (5 个 tickTime)
    • LF 同步通信时限
  • dataDir
    • 默认存储目录
  • clientPort = 2181
    • 客户端连接端口

# 服务端选举机制

# 选举时机

Zookeeper 的集群处于一下两种情况时，会发起选举:

  • 服务器初始化启动
  • 服务器运行期间无法和 Leader 保持连接

# 集群初始化时的选举流程

  1. 服务器启动时 会触发选举

  2. 每台服务器会默认先投自己一票 (选不出来时 会保持为 looking 状态)

  3. 服务器之间会交换选票信息 (携带 myid)

  4. myid 小的服务器会将选票改投给比自己大的服务器

  5. 某服务器得到选票数超过集群一半以上，选举为 Leader, 状态改为 leading; 其余服务器由 looking 状态改为 following 状态

    • 不是 looking 状态后，不会更改选票信息将自己的票再投给别人

  6. 一旦有一个 Leader , 其它服务器的选票都投给自己，无法超过 leader 服务器，自然成为 Follower,

    状态改为 following

# 运行期间选举流程

运行时发起的选举，集群 Leader 可能处于两种状态:

  1. 集群中 ** 存在 Leader (** 发起选举的服务器会和其他服务器进行通讯，通讯时会共享当前集群的 Leader 信息)

    • 发起选举的机器 hi 尝试和 Leader 服务器建立连接，并进行状态同步

  2. 集群中不存在 Leader (Leader 服务器挂掉)

    • 根据 SID (服务器 id, 和 myid 一直) , ZXID (事务 id) , Epoch (每个 Leader 任期代号) 三个值来确定

      • Epoch 大的直接胜出
      • Epoch 相同，ZXID 大的胜出
      • Epoch&&ZXID 相同，SID 大的胜出

      ===> 优先级: Epoch > ZXID > SID

# 2 客户端命令行操作

# 进入客户端

# bin/zkCli.sh -server ip:port

进入 zk 安装目录下的 bin 在指定 ip 的 port 端口启动客户端

# 退出客户端

# quit

# 查看节点

# ls path

  • 列出指定 path 下的文件节点

# ls path -s

​ 会更详细的列出相关节点信息

  • czxid : 创建节点的事务 zxid
  • ctime : znode 被创建的毫秒数 (从 1970 开始)
  • mzxid : 最后更新的事务 zxid
  • mtime : znode 最后修改的毫秒数
  • pZxid : znode 最后更新的子节点 zxid
  • cversion : znode 子节点变化号。子节点修改次数
  • dataversion : znode 数据变化号
  • aclVersion : znode 访问控制列表变化号
  • ephemeralOwner : 如果是临时节点，这个是 znode 拥有者的 session id . 如果不是临时节点则是 0
  • dataLength : znode 的数据长度
  • numChildren : znode 子节点数量

# 节点类型 & 节点操作

Zookeeper 的节点可以从两个维度进行区分:

  1. 节点的持久性 (持久 / 短暂)
    • 客户端和服务端断开连接后，创建的节点是否删除
  2. 节点的顺序序号 (有序号 / 无序号)
    • 创建 znode 时设置顺序标识，znode 名称后悔附加一个值，顺序号是一个单调递增的计数器，由父节点维护
    • 注：在分布式系统中，顺序号可以被用户为所有的事件进行全局排序，这样客户端可以通过顺序号推断事件的顺序

===> 两个维度可以将节点分为 :

  • 持久有序号
  • 持久无序号
  • 短暂有序号
  • 短暂无序号

# 创建永久节点 (无序号)

# create /path "value"

创建 /path 路径节点，节点中的值为 value

# create /path/subpath "value"

创建 /path 下的 /subpath 子节点，子节点值为 value

==> 相当于多目录创建

# 查看节点的值

# get -s /path

获取 path 节点的 znode 值

# 创建永久节点 (有序号)

# create -s /path/subpath "value"

创建 /path/subpath 下的永久有序节点 (会自动对 subpath 添加序号)

---> 终端会显示 : Created /path/subpath0000000000

# 创建临时节点 (无序号)

# create -e /path/subpath "value"

# 创建临时节点 (有序号)

# create -e -s /path/subpath "value"

# 修改节点的值

# set /path/subpath "value"

# 删除节点

# delete /path/subpath

删除指定的子节点 (必须是没有下级子节点的节点)

# deleteall /path

删除指定节点下的所有子节点数据

# 监听器流程

main 线程中创建 zl 客户端，在客户端创建两个线程 (负责网络通讯的 connet 和 负责监听的 listener). connect 线程会将注册的监听事件发送给 Zookeeper, Zookeeper 会将注册的监听事件添加到监听器列表中；

Zookeeper 监听到右数据或者路径的变化，就会讲这个消息发送给 listener 线程 (listener 线程内部调用了 process () 方法 )

# 常见的监听使用

# 监听节点数据变化

get -w /path

注册监听 /path 的节点数据变化

注：注册一次，监听一次数据的变化。当注册一次，数据变化多次时，只有第一次被监听到 (多次监听变化需要重新注册监听)

# 监听节点数量变化

ls -w /path

注：同节点数据的监听，注册一次监听一次

# 3 写数据流程

# 请求直接发给 Leader 节点

  1. 客户端向 leader 节点写入数据
  2. leader 依次向 follower 节点通知写入数据
  3. 得到写数据指令的 follower 节点写入数据
  4. 写入完成后向 Leader 节点发送 ack 响应写操作成功
  5. Leader 节点判断是否超过半数 follower 节点写入成功
  6. 超过半数 follower 节点写入成功，则直接 ack 响应客户端
  7. 其他未写入的 follower 节点完成写入

Zookeeper 认为超过半数即可应答客户端写数据响应

# 请求发送给 Follower 节点

  1. follower 节点没有写入权限，将写入请求转发到 Leader 节点
  2. Leader 节点完成数据的写入，并将写入操作下发 follower 节点
  3. follower 节点将写入完成后的 ack 响应到 Leader 节点
  4. 写入达到半数后，Leader 节点将 ack 告知响应客户端的 follower , 表示写入已完成
  5. 接收请求的 follower 将写入完成的 ack 响应到客户端
  6. Leader 将写操作下发到其他未写入的节点完成写操作
