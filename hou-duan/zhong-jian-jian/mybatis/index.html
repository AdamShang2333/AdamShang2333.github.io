# SqlSessionFactory

  1. 是用来创建 SqlSession 对象的 ，而 SqlSession 用来操作数据库的
  2. SqlSessionFactory 是线程安全的，它一旦被创建，应该在应用执行期间都存在。在应用运行期间不要重复创建多次，建议使用单例模式。
  3. SqlSessionFactoryBuilder 则可以从 XML 配置文件或一个预先定制的 Configuration 的实例构建出 SqlSessionFactory 的实例.

# SqlSession

Mybatis 中的核心
Mybatis 工作最重要的 java 接口，通过这个接口来执行命令，获取 mapper 以及管理事务
sqlSession，顾名思义，就是 sql 的一个会话，
在这个会话中发生的事不影响别的会话，如果会话提交，则生效，不提交不生效。
　 　---> 即: 如果在事务里，则 Spring 给你的 sqlSession 是一个；
　　　　否则，每一个 sql 给你一个新的 sqlSession , 且每次都会提交

以下是部分设置，标记高亮的是常用设置:

设置名 描述 有效值 默认值
cacheEnabled 全局地开启或关闭配置文件中的所有映射器已经配置的任何缓存。 true / false true
lazyLoadingEnabled 延迟加载的全局开关。当开启时，所有关联对象都会延迟加载。 特定关联关系中可通过设置 fetchType 属性来覆盖该项的开关状态。 true / false false

# Mybatis 数据库缓存

关于 Mybatis 数据库的缓存
缓存的目的 :
避免频繁的与数据库进行交互，尤其是在查询越多、缓存命中率越高的情况下， 使用缓存对性能的提高更明显

mybatis 也提供了对缓存的支持， 分为一级缓存和二级缓存。
在默认的情况下， Mybatis 开启一级缓存 (SQLSession 缓存)
一级缓存是对同一个 SqlSession 而言的，二级缓存对不同的 SqlSession 而言。

一级缓存 (SqlSession 级别):

1 同一个 SqlSession 对象，调用同一个 <select> 标签，只有第一次执行 SQL 语句，
　　第一次的查询结果会缓存到 SQLSession 的缓存区 (内存) 中
　　---> 所以被查询的实体类要实现序列化的接口以便缓存
　　有效范围：同一个 sqlSession 对象
2 缓存的是 Statement 对象 (一个 <select> 对应一个 statement )
　　也就是说，即使 SQL 语句一样 有可能也不会走缓存，只看 <select> 标签的 id 值

二级缓存 (SqlSessionFactory 级别):

1 二级缓存和一级缓存使用的是不同的内存空间，
　只有在 SQLSession.commit () 或者 sqlSession.close () 的时候，
　才会从一级缓存放入二级缓存中
　有效范围：同一个 SqlSessionFactory 中的任一 sqlSession 对象
2 当数据频繁被使用，很少被修改的时候，适合开启二级缓存， 频繁修改一定不要开二级！！
　数据被修改的时候，缓存在内存中的数据没有及时同步，会产生 "落后" 的脏数据
　那么再次查询时，读取缓存中的数据与数据库中的就会不一致

开启二级缓存的方式

  1. 在 xxxmapper.xml 中添加 <cache readOnly = "true"/>
  2. 在 xxxmapper.xml 中添加 <cache/> ; 并且结果集对应实体类实现序列化接口

# typeAliases (别名) 配置

为 Java 类型设置一个短的名字。
它只和 XML 配置有关，存在的意义仅在于用来减少类完全限定名的冗余

常用的有两种配置方式:
1 精确的指定别名
2 扫描指定包下的所有 bean

在没有注解的情况下，会使用 Bean 的首字母小写的非限定类名来作为它的别名；
若有注解，则别名为其 javaBean 类上的注解值 (@Alias ("author") )

# typeHandlers (类型处理器) 配置

typeHandler 一般在两个地方会用到:

  1. 由 java 向 DB 传入参数时 (java 类型 -->JDBC 类型 )
  2. 将结果集中的 JDBC 数据类型转换成 java 中的对应类型

实例 设置一个 Money 类的 typeHandler
java bean

public class Product {
private Integer id;
private String name;
private String currencyUnit;
private Money money;
}
public class Money {
private long value;
}

>mapper.xml
>```xml
><!--设置好Money类的typeHandler后,可以直接以#{money}的方式传入参数-->
><!--没设置的话,只能以#{money.value}的方式传入-->
><insert id="add">
        insert into tb_product 
          values (null,#{name},#{currencyUnit},#{money})
</insert>
>```
>mybatis-config.xml  
>>在配置mybatis的xml中设置typeHandler   
>**注意, 和 typeAliases一样 , package要在 typeHanlder后面**
>
>```xml
><typeHandlers>
        <typeHandler handler="com.lanou.demo.handler.JodaMoneyTypeHandler"/>
        <!--和配置typeAliases一样,也可以直接扫描包-->
        <package name="com.lanou.demo.handler"/>
</typeHandlers>
>```
>MoneyTypeHandler
>>继承BaseTypeHandler,重写其中的setxxx和getxxx的方法  
>>两种方法就是在对应向DB传入参数和取出参数
>
>```java
/**
 * 继承BaseTypeHandler (TypeHandler的一个子类)
 * 所填入的泛型,是要处理的类型(java中哪个类型的变量 要对应jdbc中的数据类型)
 */
public class MoneyTypeHandler extends BaseTypeHandler <Money>{
    /**
     * 设置传入参数时, java中数据类型与jdbc中数据类型的对应关系
     * JDBJ中设置传入参数的方法:
     * prepareStatement ps = conn.getPrepareStatement("insert into xxx values (null,?,?)")
     * ps.setString(1,aaa);
     * ps.setLong(2,bbb);
     * ****************************************
     * @param ps prepareStatement sql语句编译预处理对象
     * @param i 要处理的参数是传入的第几个
     * @param parameter 要交给类型处理器处理的属性参数
     * @param jdbcType 对应的jdbc类型
     * @throws SQLException
     * 设置好后, 在mapper.xml中使用参数 不用按照#{price.value}的形式来使用参数 可以直接#{price}了
     */
    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, Money parameter, JdbcType jdbcType) throws SQLException {
        ps.setLong(i,parameter.getValue());
    }
    /**
     * @param columnName 列名(方法中指定了泛型Money的列)
     * @return 返回对应处理的java类型
     */
    @Override
    public Money getNullableResult(ResultSet rs, String columnName) throws SQLException {
        long value = rs.getLong(columnName);
        Money money = new Money();
        money.setValue(value);
        return money;
    }
    /**
     * @param columnIndex 第几列(方法中指定了泛型Money的列)
     * @return 返回对应处理的java类型
     */
    @Override
    public Money getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        long value = rs.getLong(columnIndex);
        Money money = new Money();
        money.setValue(value);
        return money;
    }
    /**
     * 在存储过程中使用,与上面的用法基本一致
     * @param cs 可以看做prepareStatement来使用
     * @param columnIndex 第几列
     * @return
     */
    @Override
    public Money getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        long value = cs.getLong(columnIndex);
        Money money = new Money();
        money.setValue(value);
        return money;
    }
}

# environments (环境) 配置

MyBatis 可以配置成适应多种环境，这种机制有助于将 SQL 映射应用于多种数据库之中， 现实情况下有多种理由需要这么做。
例如，开发、测试和生产环境需要有不同的配置；
或者想在具有相同 Schema 的多个生产数据库中 使用相同的 SQL 映射。
不过：尽管可以配置多个环境，但每个 SqlSessionFactory 实例只能选择一种环境。

指定加载环境 :

xml 中配置加载环境:

注意这里的关键点:

  1. 默认使用的环境 ID（比如：default="development"）。
    至少要有一个 <environment> 的 id 值与 <environments> 的 default 值匹配
  2. 每个 environment 元素定义的环境 ID（比如：id="development"）。
  3. 事务管理器的配置（比如：type="JDBC"）。
  4. 数据源的配置（比如：type="POOLED"）。

===

transactionManager (事务管理器)

MyBatis 中有两种类型的事务管理器（也就是 type=”[JDBC|MANAGED]”）
JDBC – 这个配置就是直接使用了 JDBC 的提交和回滚设置，它依赖于从数据源得到的连接来管理事务作用域。
MANAGED – 这个配置几乎没做什么。它从来不提交或回滚一个连接，而是让容器来管理事务的整个生命周期（比如 JEE 应用服务器的上下文）。 默认情况下它会关闭连接

提示:
如果你正在使用 Spring + MyBatis，则没有必要配置事务管理器，
　因为 Spring 模块会使用自带的管理器来覆盖前面的配置。

dataSource type (数据源类型)

许多 MyBatis 的应用程序会按示例中的例子来配置数据源。
虽然这是可选的，但为了使用延迟加载，数据源是必须配置的.

有三种内建的数据源类型（也就是 type=”[UNPOOLED|POOLED|JNDI]”）：

  • UNPOOLED

这个数据源的实现只是每次被请求时打开和关闭连接。
虽然有点慢，但对于在数据库连接可用性方面没有太高要求的简单应用程序来说，是一个很好的选择。
不同的数据库在性能方面的表现也是不一样的，对于某些数据库来说，使用连接池并不重要，这个配置就很适合这种情形

  • 配置属性 :
    driver – 这是 JDBC 驱动的 Java 类的完全限定名
    （并不是 JDBC 驱动中可能包含的数据源类）。
    url – 这是数据库的 JDBC URL 地址。
    username – 登录数据库的用户名。
    password – 登录数据库的密码。

  • POOLED

这种数据源的实现利用 “池” 的概念将 JDBC 连接对象组织起来，
避免了创建新的连接实例时所必需的初始化和认证时间。
这是一种使得并发 Web 应用快速响应请求的流行处理方式。

  • JNDI

这个数据源的实现是为了能在如 EJB 或应用服务器这类容器中使用，
容器可以集中或在外部配置数据源，然后放置一个 JNDI 上下文的引用。
这种数据源配置只需要两个属性 : initial_context , data_source

# XML 映射 (XXXMapper.xml)

最常见和使用的就是 xml 形式的结果集映射，也可以做注解形式的配置
但是 sql 语句过长和复杂的时候，代码的可读性会下降

  • 注解形式，在方法上面使用 @select/@insert 注解就行，value 值填入 sql 语句
  • XML 形式，使用 select /insert/update /delete/resultMap 标签来完成映射

xml 映射文件结构

xml 映射文件中的配置

  • 主键返回 (insert 和 update)
  • 缓存刷新 (使用默认值即可，不用配置)

常见的结果映射可能会出现非基本数据类型，

或者某个属性是一个集合 (具有泛型): private List<Role> roles;

这种时候，需要使用 <resultMap> 来做高级映射 :

# <if> 标签

做判断用，符合条件的才会执行 <if> </if> 中包裹的 sql 语句
注意 :

  1. if 中的 test =" " 使用的是 OGNL 表达式进行的判断
    　　----> 可以直接写传入参数的 Key 或者传入对象的属性名 不需要添加任何特殊符号
    　　即：不要像 mybatis 中那样，使用 #{ } 或者 $
  2. test 中要使用 and 和 or , 不要使用 && , ||

# <choose> , <when> , <otherwise> 标签

只要有一个标签中的内容成立，下面其他的 sql 语句都不会被执行

<choose>
    <where test = " ">
    </where>
    <where test = " ">
    </where>
</choose>
