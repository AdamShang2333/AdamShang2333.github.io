# Spring 与 SpringMVC 的关系

SpringMVC 是基于 Spring 的

Spring 是一个容器技术
容器里装的是对象，他能帮助我们来管理对象
一般情况下，我们需要告知 Spring 如何来创建对象 (使用 xml 或者注解的方式)
然后从 Spring 容器中取出对象
　---> 默认情况下，取出的是单例的对象
由 Spring 容器管理的对象，叫做 bean
SpringMVC 是对 Servlet 进行的封装，是 Spring 中的一个功能模块
它的作用是可以用更简单的 API 来接收客户端发起的请求，
同样，简单的给用户返回响应

# Spring 容器和 SpringMVC 容器的关系

Spring 容器和 SpringMVC 是父子容器的关系 (Spring 包含 SpringMVC)
因此，SpringMVC 可以调用 Spring 容器中的 bean

##Spring 的核心技术

###IOC 控制反转

把创建对象的控制权交给 Spring 容器
在 Spring 中，管理 bean 的是 ApplicationContext , 是 BeanFactory 的一个实现类
默认情况下，当配置文件 (.xml) 被加载的时候，注册在容器中的类就会被创建
创建的类会由 ApplicationContext 来管理
注:
默认情况下，Spring 创建的 bean 都是单例的 **(默认 scope=singleton),
要修改创建方式，可以给 bean 配置 scope 属性修改为 prototype**

# Spring 创建 bean 的三种方式

通常有三种方法创建 bean:

  1. 通过构造方法创建
  2. 通过静态工厂创建
  3. 通过实例工厂创建
# 构造方法创建 bean

1. 通过构造方法创建
　 1.1 无参构造创建：默认状况，加载 xml 文件时会调用该类的无参构造方法
　1.2 有参构造创建：需要明确配置，在 <bean> 标签中使用开放标签，
eg: 添加 <constructor-arg index="0" value="张三"/>
　 index: 参数下标
　 name: 参数名字 (name="name")
　 type: 参数类型
　 ** value/ref: 构造方法参数值 **
注意:
1). 可以添加多个索引来定位构造方法 (index,name,type 结合使用)
2.) 如果索引无法区分参数列表长度相同的构造方法，
　　默认执行的是最后一个构造方法

# 实例工厂

工厂设计模式 (实例；抽象；静态；简单等等):
帮助创建类对象，一个工厂可以生产多个对象

实例工厂：需要先创建工厂对象，才能生产对象

步骤:

  1. 创建一个实例工厂对象
public class PersonFactory {
    public Person newInstance(){
        return new Person();
    }
}
  1. 在 spring - * .xml 中配置工厂对象和需要创建的对象
# 静态工厂

不需要创建工厂，可以快速创建对象
与实例工厂类似，只是创建对象的方法改为静态方法，由类来调用

###DI 依赖注入

由 Spring 容器来为 bean 的属性赋值，这就叫做依赖注入

注解注入会先于 xml 注入执行 ——》因此，xml 注入会覆盖注解注入
如果是自己写的类，建议使用注解（注解需要改源代码）
如果是系统类或者框架内部的类， 一般只能使用 xml 配置

# xml 配置

spring 的配置是基于 schema (xml 的语法检查器) 的，它的扩展名是 **.xsd**

xmlns: xml 的命名空间 (nameSpace)
每个 xmlns 都要有一个与之匹配的.xsd 文件，才会有相应模块的代码提示和检查
==> 在 xsi:schemaLocation="xxx.xsd" 中体现

# SpringWeb 项目的配置入口 --web.xml

整个项目最关键的配置是 web.xml 文件 ---> 配置过滤器，监听器和 servlet
其次，是各种 spring--*.xml 的文件 ---> 配置组件的扫描 (注解), 依赖注入等...

Spring 的关键是创建了一个 DispatherServlet 对象 对所有的请求做统一的处理和分发；
而关键的 JavaWeb 三大组件:
　--->Servlet; Filter ; Listener
需要在 web.xml 中进行配置，配置的顺序有优先级: listener -> filter -> servlet

注意 web.xml中 :

  1. 对于不同的配置节
    加载顺序与它们在 web.xml 文件中的先后顺序无关。
    ---> 即不会因为 filter 写在 listener 的前面而会先加载 filter.
  2. 对于相同的配置节
    加载的顺序与他们的出现顺序有关
    ---> web 容器启动时初始化每个 filter 时，是按照 filter 配置节出现的顺序来初始化的；
    　　当请求资源匹配多个 filter-mapping 时，
    　　filter 拦截资源是按照 filter-mapping 配置节出现的顺序来调用 doFilter () 方法的
  3. 对于相同配置节中的映射顺序
    filter-mapping 必须出现在 filter 之后，
    否则当解析到 filter-mapping 时，它所对应的 filter-name 还未定义。

####Servlet 配置的方式

配置有两种方式来实现:

  1. 在 Servlet 类中使用基于注解的方式进行配置：@WebServlet
  2. 在 web.xml 文件中进行配置。
    　---> 一般使用这种方法配置 (可以更好的提高代码的扩展性，便于修改维护，解耦)
# Web.xml 基本配置实例

对应在Servlet中管理的bean的配置文件--spring-config.xml

组件扫描 :
实际上就是在扫描指定包路径下的所有类，然后把一些类直接配置为 bean

可以被组件扫描的类 : **
添加了如下注解的类，可以被组件扫描
@Controller 　　如果一个类是 Controller** , 那么就用 Controller 注解
@Service 　　　 如果一个类是 Service , 那么就用 Service 注解
@Repository 　 如果一个类属于 DAO 层，那么就用 Repository 注解
@Component 　如果一个类不属于上述任何一个类别 那么就使用 Component 注解
注:
实质上这几个注解没有实质上的区别 都会配置为 bean
只是为了在语义上有所含义，提高代码的可读性

# DI 和 IOC 的关系

Ioc 的过程，是将 bean 注册到 Ioc 容器，让 Ioc 容器进行 bean 的管理过程；
在注册的过程中，需要对 bean 的属性进行赋值.
如果对 A 赋值的属性是另一个对象 B, 那么 A 就依赖于 B
通过 xml 描述 A&B 的依赖关系的过程就是 DI
即: Ioc 和 DI 在本质上是在做一件事情，在 Ioc 的过程中会伴随着 DI
