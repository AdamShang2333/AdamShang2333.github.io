## 代理模式

代理模式给某一个对象提供一个代理对象，并由代理对象控制对原对象的引用。
通俗的来讲代理模式就是我们生活中常见的中介。

### 代理的作用
中介隔离作用 ：
在某些情况下，一个客户类不想或者不能直接引用一个委托对象，而代理类对象可以在客户类和委托对象之间起到中介的作用，其特征是代理类和委托类实现相同的接口。

开闭原则，增加功能 ：
代理类除了是客户类和委托类的中介之外，我们还可以通过给代理类增加额外的功能来扩展委托类的功能，这样做我们只需要修改代理类而不需要再修改委托类，符合代码设计的开闭原则。
代理类主要负责为委托类预处理消息、过滤消息、把消息转发给委托类，以及事后对返回结果的处理等。代理类本身并不真正实现服务，而是同过调用委托类的相关方法，来提供特定的服务。真正的业务功能还是由委托类来实现，但是可以在业务功能执行的前后加入一些公共的服务。
例如我们想给项目加入缓存、日志这些功能，我们就可以使用代理类来完成，而没必要打开已经封装好的委托类。

### 分类

按照 代理创建的时期 来进行分类的话，
可以分为两种：静态代理、动态代理。
静态代理 是由程序员创建或特定工具自动生成源代码，在对其编译。在程序员运行之前，代理类.class 文件就已经被创建了。
动态代理 是在程序运行时通过反射机制动态创建的。

按照 == 实现方式 == 来进行分类:
可以分为：通过接口实现，通过继承实现
　注：在动态代理中，通过 JDK 方式 (接口) 或者 CGLib (继承) 来实现动态代理

####Step2 委托类，实现服务类接口

创建代理类，有两种方式:

  1. 通过实现与委托类相同的接口
  2. 通过继承委托类
    　 继承最好也要引入父类作为成员变量

3.1 代理类实现与委托类相同的接口

3.2 代理类继承委托类
直接调用父类方法

传入父类作为成员变量

public class LogCategoryService extends CategoryService {
    // 将父类作为成员变量传入
    private CategoryService categoryService;
    public LogCategoryService(CategoryService categoryService) {
        this.categoryService = categoryService;
    }

    @Override
    public void del() {
        System.out.println("开始删除");
        // 调用成员变量方法
        categoryService.del();
        System.out.println("删除分类");
    }
}

两种方式都可以通过继承来实现代理，但是传入父类作为成员变量更好些

#### 测试

继承实现与接口实现的区别
继承方式：每多一个逻辑，多一个代理类；逻辑顺序改变一次顺序，多一个代理类
接口方式：每多一个逻辑，多一个代理类；逻辑顺序改变一次顺序，只需在调用时改变顺序

## 动态代理

多个方法需要代理的时候，使用静态代理就需要多次编写代理类和代理逻辑
在逻辑重复的场景下，比较繁琐 (日志打印 / 登录验证等)
此时适合使用动态代理

# 实现方式

  1. 使用 JDK 反射包下的动态代理类进行实现
    针对有接口的类进行动态代理
  2. 使用第三方的框架 Enhancer (jar 包:CGLib) 实现动态代理
    针对可继承的类进行动态代理

# 1. JDK 动态代理实现 (接口实现)

步骤 :
1 需要 new 一个代理实例 (通过 JDK 自带的 Proxy.newProxyInstance () 方法来实现 )
　参数：类加载器 ；委托类实现的所有接口；具体代理方式 (逻辑)　
2 获取类加载器
　获取类加载器的方式有很多，通过任意一个类来获取都可以:
　　 bs.getClass().getClassLoader()
　或者通过 ClassLoader 类的 getClassLoader 方法获取
　　 ClassLoader.getSystemClassLoader()
3 获取委托类的接口类型
4 传入一个具体代理方式 (实现 InvocationHandler 接口)

InvocationHandler 接口是 proxy 代理实例的调用处理程序实现的一个接口，
每一个 proxy 代理实例都有一个关联的调用处理程序；
在代理实例调用方法时，方法调用被编码分派到调用处理程序的 invoke 方法。

自定义的 InvocationHandler 实现类

###2. CGLib 动态代理实现 (没有接口)

CGLib 的动态代理和静态代理中的继承方式类似
也需要让代理类继承一个委托类
创建代理类的时候，和 JDK 一样 也需要一个代理逻辑实现类
这个类实现一个 InvocationHandler 接口 (CGLib 包下，该接口 extend CallBack)

##AOP

AOP 是 Spring 的核心技术 和 SpringMVC 无关
AOP, 面向切面编程 本质上就是
　 对哪些方法执行的什么时候，做什么事情
--"对所有 Service 方法执行之前，打印日志";"对所有的 Service 方法，执行异常的时候，回滚事务"
AOP 的实现原理是动态代理，会根据实现方式的是接口还是继承
　来自主的选择 CGLib 或者 JDK 反射

# AOP 的组成部分

  • 切点 (PointCut) :
    实际上是一个表达式 (也就是一个字符串), 用来表示对哪些方法进行增强
    切点实际上描述的就是 "哪些方法"
  • 通知 (Advice):
    实际上就是方法，表示如何进行增强
    通知实际上描述的是 "什么时候" 和 "做什么事情"
  • 前置通知 在执行方法之前
  • 返回通知 正确执行，返回执行之前
  • 异常通知 抛出异常时执行
  • 后置通知 无论方法是否正常执行，都会被调用
  • 环绕通知 综合了上面的四种通知
  • 接入点 (JoinPoint):
    实际上就是被增强的方法的一个描述
  • 切面 (Aspect):
    把切点 & 通知 & 接入点封装起来的一个类
  • 织入 (Weaver):
    把切面整合到原始对象的过程
  • 增强 ( ):
    注:
    Spring 利用了 AspectJ 的切面的配置方式 (注解设计), 具体实现没有使用它的，
    而是使用了 JDK 中带的和 CGLib 框架中的动态代理

### 配置

具体配置有两种:
一个是在 xml 中进行配置；一个是使用注解
不管用哪种，配置项都是一样的，只不过具体的方式不一样.

# xml 配置

#### 注解配置

注解方式的配置只需要在 xml 中添加一个注解支持就行

  • 在 xml 中设置对注解的支持

使用注解形式时，对切面类的配置内容与 xml 配置一样
只是部分内容不同:

  1. 需要在切面类中添加一个无方法体的方法，添加 @PointCut(execution( )) 注解
    　　-- 为了调用到注解内的切点表达式 (注解只能标注在类或者方法)
    　　　　　注意：方法的名称就是切面类 配置通知标注时的填入的值
    　　　　　常用切点表达式: * 包名。类. 方法.(参数)
  2. 在类上添加 @Component & @Aspect 注解标识为切面类
  3. 在切面类的通知方法上添加对应的通知类型注解，
    注解内填入该通知生效的切点 (哪些方法被增强)
    　各类型通知的注解 :
    @Before ("allService ()") // 前置通知
    @After ("allService ()") // 后置通知
    @AfterThrowing (value = "allService ()",throwing = "ex") 异常通知
    @AfterReturning (value = "allService ()",returning = "value") 返回通知
    @Around ("allService ()") 环绕通知

切点表达式
execution 的语法表达式如下：
execution (<修饰符> < 返回类型 > < 类路径 > < 方法名 >(< 参数列表 >) < 异常模式 > )
