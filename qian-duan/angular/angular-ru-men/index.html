# 父传子

  • @input（）
    • 可以标记到任何属性值上，代表该属性由父组件传入
    • 需要在对应插入子组件的 html 中 使用 [field] = ‘obj.value’ 这种形式来将数据传输
    • [ ] 是用来绑定属性值的 ， 对应的传值使用单引号
    • ( ) 是用来绑定方法的 ，对应的传值使用双引号

# 子传父

  • @output( )
    • ！！该装饰器函数与 @input 不同，绑定的属性必须是一个事件
      • 原因：嵌套组件将数据传递回父容器的唯一方法是发出事件
      • 事件通过 EventEmitter< type > 对象定义
      • 子组件中传递的数据 需要与声明时的范型保持一致 ， 并通过 **EventEmitter.emit (value)** 来传递给父组件
// 子组件中声明传递出去的事件对象，事件名称
@output( ) notify : EventEmitter<string> = new EventEmitter<string>( );

// 父组件的 html 中 , 对子组件的事件进行绑定
// （notify): 指定父组件要监听的事件名称（事件名称声明在子组件中）
// ‘onNotify($event)’ :  将事件返回值通过$event 声明 ， 并传递到父组件中的onNotify( )方法
// 父组件中的 onNotify( )方法的入参类型 ，就是子组件在声明 EventEmitter 时传入的范型
<p-star (notify) = ‘onNotify($event)’>

// 父组件的 ts 中 获取对应的事件对象
onNotify(msg: string): void { }

# Service 服务

service 服务的初衷， 是避免使用组件创建服务类的实例并使用
但是该实例是组件的本地实例， 我们无法共享数据或其他资源。mock 测试也更困难 ，所以使用依赖注入来交由 Angular 管理 service

# 服务和依赖注入的概念

  • 服务类交由 Angular 创建（一般是单例，根据需求也可创建多例）
    • 由于 Angular 管理单个实例，因此使用该实例的所有类都共享该实例中的任何数据或逻辑
  • Angular 提供内置的注入器 ， 我们向其注册服务。该注入器维护一个已创建服务实例的容器
  • 组件类中声明对 service 服务的依赖 ， 即可交由 Angular 注入对应的服务类实例
    • 当 Angular 实例化组件类时，Angular 注入器将提供或注入服务类实例 ， 这个过程就称为依赖注入

# 构建服务的步骤

  1. 创建服务类 （创建 class）
  2. 使用装饰器定义元数据 （@Injectable（）修饰类）
  3. 导入需要的模块 （import {Injectable} from ‘@angular/core’）

与创建模块的步骤类似，主要区别是创建并注入的时候， 有 root Injetor ， component Injector 两种注入器。

每个组件的注入器，可以理解为对组件树结构的镜像

  • Root Inject
    • 该注册器中注册的服务， 可用于应用中的任何组件或其他服务
  • Component Inject
    • 向特定组件注册的服务，仅可用于该组件以及其子组件或嵌套组件
    • 组件中注册的服务是多例的， 每一个组件的实例会对应一个相应的服务实例

# 注册服务

服务的注册一般都是使用 root Inject 即可：

更建议的方式是使用 providedIn， 这样可以更好的 “摇树”。
---> 摇树是一个过程， 编译器通过该过程摇晃未使用的代码，以部署较小的捆绑软件

# 服务的使用（如何进行依赖注入）

与 Java 类似，每一个类都有一个构造函数，该构造函数在创建类实例的时候执行。 如果没有显式定义构造函数，则会使用隐式构造函数。

注意： 如果要注入依赖项，则需要显式的构造函数！！

关于构造函数：

  • 构造函数会在创建组件初始化的时候执行
    • 不适合在构造函数中使用异步回调的函数执行
    • 构造函数应该 “纯净”，不会产生对其他部分的影响
  • 构造函数的参数标识组件的依赖关系
    • 一般会创建一个私有变量来保存注入的实例（该实例在构造函数中来进行赋值）

# HTTP 请求

http 请求的实现，通过订阅可观察对象并获取视图数据

# RxJS 简述（ReactiveExtensions）

Observable 对象
可以简单的理解为一个数据流，需要订阅事件并定义异步执行的方法

# 常见的通知类型

订阅之后，接收到 notify 会有以下三种类型

  • next
    • next item is emitted
  • error
    • an error occurred and no more items are emitted
  • complete
    • no more items are emitted

# Observable 的常用函数

# Pipe

pipe 方法中可以传入若干个操作函数，以逗号分隔。
操作函数的入参一般是一个箭头函数 ，或一个引用的函数的函数名（this.functionName ）

  • tap
    • 在不变换流中数据的情况下， 查看其中的数据
    • 非常适合调试或日志记录
  • filter
    • 过滤数据流
  • map
    • 将传入的数据根据映射函数做转换

  • # mergeMap

  • catchError
    • 捕获任何的错误
    • 处理 Http 异常时，需传入一个 HttpErrorResponse 对象
# subscribe

使用上和 pipe 函数类似， 可以传入 function，来处理订阅到数据后的操作逻辑

  • x.subscribe( )
  • x.subscribe( observer )
  • x.subscribe({
    next: nextFn,
    error: errorFn,
    completeFn
    })
    • next ，error ，complete 是三种不同类型的通知消息
    • 以键值对的形式给出函数的定义，（next 中的参数，即是 observable 对象范型中声明类型的 item，next 函数指定了当可观察对象发出下一个值的时候，需要做什么处理）
# unsubscribe

取消订阅。 一般会在 onDestroy 函数中

# http 请求的执行流程概述

  1. 调用请求
  2. 返回一个 Observable 对象（会发送通知事件，程序可以知道响应何时到达）
  3. 订阅 Observable 对象以启动它 （在订阅之前， Observable 不会做任何事情）
  4. 等待响应
  5. Observable 发出通知， 程序处理发出的响应

# Observable 对象的常见使用流程

常见的使用流程如下：

  • 通过订阅启动 Observable 对象（subscribe）

    • 实际代码上，是在第二步的 pipe 转换之后再通过 subscribe 方法，操作管道处理完成的数据流

  • 通过 pipe 管道传递并转换 item （pipe function）

    • 可以在 pipe 方法中添加任意数量的 operator 方法顺序执行

  • 再处理 Observable 对象的 notify （next ， error， complect）

  • 取消订阅事件 （unsubscribe）

    • 一般取消订阅放在组件生命周期结束的时候

# Route 简介

Angular 是一个单页面的应用，这意味着所有组件对应页面需要根据需求可控制的出现在 index.html 中。
管理何时显示哪一个视图，就是路由的目的。

路由是基于组件的，在配置路由时 ，实质上是配置路由路径和组件的映射关系

# 路由定义

angular 的路由器需要指定路由的路径 path，和激活此路由时需要加载的组件

# 路由的显示

路由被匹配到相应的组件后， 会展示到以内置指令 <router-outlet> 指定的位置

# 路由的配置

RouterModule 和 Router 模块，是 Angular 提供的路由功能模块

触发路由的方式

  • a 标签驱动（ <a routerlink = '/products'> ）
    • 使用 angular 的内置路由指令， 可以指明 a 标签在点击后跳转的路由
    • H5 路由 ( <a routerLink = '/products'> ==> www.web.com/products )
      • 不需要哈希符号来指示本地导航
      • 需要配置 web 服务器以执行 URL 重写
  • 哈希路由 ( www.web.com/#/products )
    • 不需要配置 web 服务器来重写 URL
  • 自定义事件驱动

路由模块的声明
路由模块必须有一个路由对象 Routes 来指明匹配规则，包含以下常用属性：

  • 路径参数使用 :parms 来指明
  • component， 指明关联 path 的组件
  • redirectTo，路由重定向指向的路径
  • children [ ] , 声明关联的子路由数组
  • loadChildren ， 懒加载子路由
  • resolve，
  • canActive [ ], 路由守卫 能否激活路由
  • canActiveChild [ ] ， 路由守卫 能否激活子路由
  • canDeactive [ ] ， 路由守卫 能否关闭路由（离开当前路由的组件）

路由匹配的注意点

  1. 路由 path 没有 / 前缀
  2. 路由器使用 "先赢" 策略， 因此路由数组中的路由顺序很重要
    更具体的路由应始终在较不具体的路由之前

# 路由参数的获取

Angular 提供的 ActivatedRoute 服务，可用作获取激活路由的参数，在构造函数中注入该服务的依赖即可

# ActivatedRoute

  • snapshot , 激活路由的快照
    • 仅需一次读取 route 参数时适用该方法
    • route.snapshot.paramMap.get('id')
  • paramMap (返回一个 Observable)
    • 需要在路径变化时随之动态获取参数时适用该方法
    • this.route.paramMap.subscribe(param => console.log(params.get('id')));

上述两种方法中传入的字符串，必须与配置路由时指定的路径参数名称一致 (path/:id)

# null 和 undefined 处理

对于 null 和 undefined 的数据， 可以使用｜来做一个属性声明 ，告诉编译器属性值可以接受为空的类型值。

上述方式对于在使用双向绑定时不会生效

product : Iproduct | undefiend
// 这种情况下的空值处理不会生效
[(ngModel)] = 'product?.productName'
<span>{{product?.product.productName}}</span>

针对上述情形，可以使用 ngIf 来对 html 和其子元素做限制，不满足时，此元素及其所有子元素都不会出现在 dom 中，因此它不会尝试检索未定义属性

# 路由跳转

路由跳转需要引入 angular 内置的 Router 服务
关键方法 ： router.navigate([pathStr])
通过该 API 来完成跳转

# 路由守卫 (Routing Guards)

路由守卫 ，就是在尝试改变当前路由路径时（in && out），添加自定义的逻辑来处理路由的操作。
eg：
仅允许特定人员的访问权限；
离开页面前的保存询问；

# 常见路由守卫

  • canActive
    • 路由进入某一 URL
  • canDeactive
    • 路由离开某一 URL
  • resolve
    • 在激活进入路由前提前获取数据
  • canLoad
    • 防止异步路由

# 路由守卫实现

  1. 路由守卫都是接口，我们自定义的路由守卫逻辑，首先需要实现相应类型的接口；
  2. 使用 @injectable 装饰器完成 service 的注册
  3. 实现接口定义的方法体

# Module 简介

# 什么是模块

angular Module 本质上是带有 @NgModule 装饰器的类。
目的是组织应用的各个部分，将他们安排在紧密结合的功能块中，并使用外部库中的功能扩展我们的 app。

  1. angular Module 为解析组件中的指令和管道提供了环境
  • Angular 模块声明了它管理的每个组件，指令，管道。我们创建的每个组件指令和管道都属于 angular 模块。它为编译器提供了一种查找和解析我们放入 HTML 的特殊 Angular 语法的方式
  • 在组件模版中使用指令等时，Angular 会在模块中查找该指令定义。如果定义该指令的组件未在同一模块中声明或未从导入的模块导出，则 Angular 将找不到该指令，并报错。
  1. 程序启动时 可以立即加载 Angular 模块；或者通过路由器异步的延迟加载
  1. module 中可以导入和导出其他的 module，
    @NGModule 中的元数据
  • imports 数组
    • 导入外部的模块，包含路由器模块
  • declarations 数组
    • 声明属于该模块的组件，指令和管道
  • bootstap 数组
    • 定义作为程序起点的组件

# 模块的元数据 (metadata)

# Bootsrtap Array

每个 Angular 程序至少有一个可称为 root module 的模块（App Module），
并且至少有一个称为 root compoment 的组件（AppComponent）。
appModule 引导 AppComponent 来提供 index.html

BootstrapArray 数组中定义的是程序起点组件，会在启动应用程序时加载。
使用时注意以下几点：

  • 引导程序数组只能在 AppModule 中使用

# Declaration Array

定义属于此模块的组件，指令和管道
注意事项：

  • 每个组件，指令和管道都必须属于且只能属于一个模块
  • 不要将其他类，服务或模块添加到声明数组中
  • 默认情况下，所有声明的组件，指令和管道都是私有的。他们只能由同一模块中声明的其他组件，指令和管道访问
    • 如果想 "打破" 默认的私有化，则需要在 export 中导出

# Export Array

导出数组允许我们与其他模块共享组件，指令和管道。

  • 任何组件，指令和管道都可以导出，以便在另一个模块使用 import 数组导入该模块时，可以将他们拉入
    • 导出的模块包含 angular 的系统模块和第三方模块，甚至是当前模块自己
  • 模块可以导出其声明的任何组件，指令和管道
    • 重新导出该模块以实现重新到处其中的组件，指令和管道。
    • 当合并多个模块功能以构建便捷模块或共享模块时，上述功能很有用。我们可以导出某些内容而无需将其包含在 import 数组中

# Imports Array

  • 通过 import 数组导入的模块 A，可以使用该模块在其 export 数组中定义的任何组件，指令和管道
  • 仅导入该模块需要的内容
  • 导入模块不存在继承
    • A 导入 B，则 A 中可以使用 B 中的任何组件 指令和管道；
    • B 导入 C，B 中同样可以使用 C 中的任何组件，指令和管道；
    • 但是 A 无法使用 C 中的组件，指令和管道（无法实现导入的继承）
    • ！！！ 注意，如果 B 中重新导出了 C，则在 A 中 时可以使用到 C 的
  • 通常，我们添加一个将服务注册到 AppModule 的 imports 数组的模块（注册 Angual 系统模块 && 第三方模块）。这样可以确保在初始化时注册一次服务

# Provides Array

提供应用程序注册服务。
我们可以使用 providers 数组向根应用程序注入程序注册我们的服务

  • 该方式从 Angular6 之后不再推荐
    • 更好的方法是使用服务本身的 providerIn 属性

# Feature Module && Shared Module

一般的模块分组优化，至少分为两个大的部分：

  • 与业务相关的功能模块（Feature Module）
  • 可复用的共享模块（Shared Module）
  • 将一组常用的组件组织到一个模块中，并导出。以便任何导入该模块的地方都可以使用他们

# Angular CLI

# 什么是 CLI

CLI 是 Angular 的命令行界面， 主要目的是帮助开发人员构建 Angular 程序，生成 Angular 文件， 构建和提供应用程序，运行测试和构建应用程序。
Angual CLI 命令可以使用 ng 来访问 ，访问指令格式如下：
ng <command> <args> --<options>
Angular 会在每个 Angular 项目的本地自动安装 Angular CLI ，这样 随着时间的推移 ，Angular CLI 的本地版本和 Angular`` 的项目版本匹配。

全局安装 Angular CLI ：
npm install -g @angular/cli

# 常用指令

  • ng new
    • 创建新的 Angular 应用
    • options ： --prefix （指定选择器前缀，自定义模块的前缀）
    • 该指令会自动创建最基础的 angular 项目结构
      • main.ts ==》 引导程序的文件，基本会不修改它
      • index.html ==》 根 HTML 页面
      • environments ==》 环境配置
      • polyfill.ts ==》 支持流行的浏览器和经典老版本浏览器
        • 默认情况被设置为支持流行的浏览器
  • ng serve
    • angular 应用运行的指令
    • 会将我们的模版和 TS 代码转换为 JS ，它从 JS 生成应用程序包，启动一个监听端口 4200 的本地 web 服务器。
    • 转换生成的主要文件如下：
      • index html (会引入其他转换的 JS 文件)
      • style.js. 应用的样式
      • vendor.js
      • polyfills.js 浏览器支持
      • main.js 主程序（如非特殊指定 则我们所有的应用程序代码都将被编译到这个包中）
    • 在内存中为应用程序提供服务，并在我们更改任何文件时重建
  • ng generate
    • 利用 "蓝图" 来生成组件 模块等的代码
    • 生成代码的模版可以自己做定制化。
    • 生成组件 (components)
      • ng g c <name>
    • 生成指令 (directives)
      • ng g d <name>
    • 生成路由守卫 (route guards)
      • ng g g <name>
    • (interface)
      • ng g i <name>
    • (modules)
      • ng g m <name>
    • (pipes)
      • ng g p <name>
    • (services)
      • ng g s <name>
    • ng test
    • ng build
      • ！！ 当我们部署到生产环境时，我们希望包尽可能的小。这意味着我们需要缩小 / 丑化代码，并执行称为 "摇树" 的操作
      • 摇树： 摇动我们的代码以删除任何 "死分支"，这基本上意味着它删除了所有未使用的部分。 CLI 就可以为我们做这件事
      • 构建完成后，项目中会生成一个 dist 文件夹。实际编译运行的 JS 文件会在这个文件夹中
